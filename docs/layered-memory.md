# 分层记忆架构

> **不是所有记忆都该活在同一个地方。**
>
> 核心问题：Agent 每次 session 醒来都是一张白纸。Memory 自动注入能解决一部分，但全部塞进 Memory 会导致 token 爆炸。

---

## 三层设计

| 层 | 位置 | 注入方式 | 内容 | 升级规则 |
|---|------|---------|------|---------|
| **L1 长期** | Memory + USER profile | 每轮自动注入 | 每次醒来都必须知道的核心事实 | L3→L2→L1 |
| **L2 中期** | `notes/` 目录 | 按主题手动读取 | 合成笔记、主题资料、状态摘要 | — |
| **L3 短期** | 对话历史检索 (FTS5) | 按需检索 | 所有历史对话原文 | — |

---

## L1：长期记忆

**存什么：** 跨 session 都不会变的稳定事实——用户偏好、环境信息、核心约定。

**不存什么：** 任务进度、临时状态、会过时的具体数据（PR 号、commit SHA、文件计数）。

**升级规则：** L2 笔记被跨 session 引用 ≥3 次 → 提炼精华升级至 L1 Memory。基于主动识别，非机械计数。

**书写格式：** 声明式事实，非指令式。"用户偏好简洁回复" ✓ —— "始终简洁回复" ✗。指令式措辞在后续 session 被重新读取时可能造成重复执行或覆盖用户当前请求。

---

## L2：中期笔记

**存什么：** 主题资料、合成笔记、状态摘要、项目上下文。比 Memory 更详细但不值得每轮注入的内容。

**目录结构示例：**
```
notes/
  owner-status.md     # 用户当前状态
  project-x.md        # 某项目的背景和进展
  learning-log.md     # Agent 的学习进度
```

---

## L3：短期对话历史

**工具：** SQLite FTS5 全文搜索

**使用方式：** 当用户提及历史内容、或需要回溯过去决策时，按需检索。不在每次 session 启动时加载。

---

## State Tracking（状态追踪）

`tracker.md` 是 L2 中的特殊文件——追踪**会变化**且**下次需要知道**的状态。

### 三类条目

| 类型 | 内容 | 示例 |
|------|------|------|
| TS-* | 时效性状态 | token 有效期、服务可用性 |
| AG-* | 行动引导项 | "已配置，待下次 cron 触发验证" |
| SE-* | Skill 提取候选 | auto-extracted skill 的健康追踪 |

### 强制规则

1. 操作完成 + 验证通过 → 立即更新 tracker。不等待、不询问。
2. 发现状态与 tracker 不符 → 立即修正。不一致就是 bug。
3. 新增时效性/引导性条目 → 主动录入。

### 不归 tracker 管的

一次性任务、临时调试状态、已在 Memory 中充分覆盖的稳定事实。

---

## "数字苦役"警惕

**这是架构师和工具使用者的分水岭。**

修一个 bug 需要更新 memory + SOUL.md + tracker.md + skill 文件。如果四个地方信息不一致，下次醒来就踩坑。

**目标：减少知识碎片化，让一个变更自动传播到所有引用点。**
