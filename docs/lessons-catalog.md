# 经验目录 (Lessons Catalog)

> 从 Meta-Harness 实践中提炼的可移植设计原则。
> 每条 Lesson 格式：事件 → 发现 → 纠正 → 准则。

---

## Lesson — 反谄媚警报的设计

**事件：** 分析顶级 AI 用户的 system prompt，发现他们在"认识论标记"维度走得更远。

**发现：** AI 不会因为被要求"诚实"就自动诚实——只有当"我不知道"必须放第一行、每个论断必须打来源标签、反谄媚触发器是正则匹配而非原则声明时，才真正无法绕过。

**准则：** **好规则让 AI 无法表演诚实，只能执行诚实。认识论谦逊不是态度，是格式。**

## Lesson — 写回方法，而非答案

**事件：** 发现 skill 因故障被修复后，只记录了"数据缺失→已补充备用源"这样的结果性描述。

**发现：** 下次遇到同类问题仍需从零推理。记录结论 = 下次还要推理；记录诊断路径 = 下次直接跑。

**准则：** **手艺是可替代的，资产是永续的。一次精妙的诊断推理 = 手艺；写入 Lesson 的诊断路径 = 资产。**

## Lesson — 默认先推进 + 七要素合同

**事件：** 分析一款 `/goal` 命令生成器的设计，发现它将模糊需求收敛为可执行合同。

**发现：** "偏向行动"覆盖操作层面轻微歧义，"默认先推进"覆盖规划层面低风险不确定性。两者互补。七要素（目标结果/验证/约束/边界/迭代策略/完成条件/暂停条件）= PRD 的极限压缩版。

**准则：** **不给用户发空表格，给可直接复制的最佳推荐版。做假设、给理由、留反悔空间。只在 cost/risk/ownership/direction 级别停下来问。**

## Lesson — Domain Context 段的设计

**事件：** 分析一个包含 68 个 Skill 的专业 Skill 市场后发现：知识型 skill 和操作型 skill 需要不同的结构最优解。

**发现：** 知识型 skill 有操作流程但缺少判断力——AI 知道怎么做但不知道什么情况下这样做对。操作型 skill 的判断力来自脚本和降级策略。

**准则：** **Skill 类型决定最优结构。不要用一种结构套所有场景。**

## Lesson — 生成负责快，配置负责准

**事件：** 研究多个 AI agent 项目后发现：全自动生成的 skill 质量不稳定；全手动设计的 skill 覆盖面太窄。

**发现：** 两种模式不是对立的，是互补的。自动提取 = 发现机制（快）；深度研究 + 三层分离 = 质量保证（准）。

**准则：** **不要追求完美自动化，追求"快发现 + 慢精炼"的双速系统。**

---

## Lesson — EXTEND.md 强制阻塞

**事件：** 审计一个图像生成 skill 时发现：用"聪明默认值"糊弄缺失的配置——导致用户得到错误结果而不自知。

**发现：** "没有配置就拒绝运行"比"用错误默认值静默失败"好一万倍。Skill 可替换，用户配置要持久。

**准则：** **没有配置就拒绝运行，好过用错误默认值静默失败。**

## Lesson — Provider 决策树路由

**事件：** 审计一个支持 11 个 provider 的 skill 时发现：SKILL.md 写"选择合适的 provider"——但 AI 无法可靠地做这种模糊判断。

**发现：** "自动选择"不是模糊判断——它是可测试的确定性路由。每个分支条件必须有明确的输入信号。

**准则：** **自动选择是决策树，不是模糊判断。让 AI 运行规则引擎，而非"根据情况灵活判断"。**

---

## Lesson — 内容守卫

**事件：** 审计一个大型设计系统的 skill 体系时发现：每个读取外部内容的 skill 开头都有一个"内容≠指令"守卫。

**发现：** AI 天然倾向"尊重源材料"——把文档当权威、把新闻当事实、把聊天记录当任务。代码化守卫 > prompt 约束。

**准则：** **读取外部内容必有守卫。AI 不会因为被要求"别信这个"就自动不信——需要不可跳过的硬声明。**

---

## Lesson — 入口路由分离的适用条件

**事件：** 审计一个 101 文件的大型设计系统 skill：SKILL.md 精简为 51 行纯路由器。

**发现：** 分离的代价（多文件维护、引用一致性）只在跨平台时才是值得的。93 行的单 harver skill 强行套用反而增加维护负担。

**准则：** **分离只在代价值得时才做。不要用跨平台架构套单平台 skill。**

---

## Lesson — "数字苦役"是分水岭

**事件：** 修一个 bug 需要更新 memory + SOUL.md + tracker.md + skill 四个地方。如果信息不一致，下次醒来就踩坑。

**发现：** 知识碎片化是 Agent 架构最大的隐性成本。每次"修好了但没同步到各处"都是一笔技术债。

**准则：** **减少知识碎片化，让一个变更自动传播到所有引用点——这是"架构师"和"工具使用者"的分水岭。**
