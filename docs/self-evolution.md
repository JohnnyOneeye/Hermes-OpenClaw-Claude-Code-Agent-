# 自进化机制

> Meta-Harness 最核心的部分——让 Agent 不只是执行任务，而是持续改进自己。

---

## 1. Extraction Triggers（成功路径提取）

复杂任务完成后，自动评估是否值得固化为 Skill。

### 触发条件

| 条件 | 说明 |
|------|------|
| 单次任务工具调用 ≥ 3 次 | 复杂度足够 |
| 跨多轮对话完成完整目标 | 跨 session 可复用 |
| 用户说"记住这个流程" | 强制触发 |

### 原则

- **宁缺毋滥。** 置信度 < 0.6 不创建
- **自动提取 ≠ 替代深度研究。** Auto-extracted skill 是草稿，后续需过 Phase 1 认知准备
- **提取是发现机制，不是质量保证**

---

## 2. Automation Reflex（自动化肌肉记忆）

> **重复 3 次以上的事，必须自动化。这不是建议，是要求。**

### 触发

- 同一操作在当前 session 已手动执行 ≥ 3 次
- 同一检查/验证跨 session 反复出现
- 任何一个"又来了"的重复性动作

### 积累效应

一个一个小的自动化攒起来，会在没反应过来时长成苍天大树。别一上来就想搭建完整自动化体系——从今天开始，花十分钟自动化掉一件重复做的事。明天再找一件。

---

## 3. 写回方法，而非答案

当任何 skill 因故障被修复后，禁止只记录修复结果。必须回答三个问题：

| 问题 | 坏例子 | 好例子 |
|------|--------|--------|
| 根因是什么？ | "数据缺失"（症状） | "单点依赖且没有预设备用源"（根因） |
| 下次按什么顺序排查？ | "要准备备用源"（结论） | "判断是否需要备用源的五步决策树"（方法） |
| 有没有可删除的旧规则？ | — | 这次故障是否证明某条旧规则无效？ |

**判断标准：** 写下的是"方法"而非"结论"。

## 4. MAPS 自检循环

每次重大改进后跑一遍：

| 维度 | 问题 | 动作 |
|------|------|------|
| **M - 心智 (Scan)** | 这暴露了什么盲区？什么问题之前看不见？ | 扫描 |
| **A - 架构 (Design)** | 我的解决方案是什么？改了什么结构？ | 设计 |
| **P - 提示词 (Dialogue)** | 具体改了什么文件、配置、规则？ | 实现 |
| **S - 系统 (Deploy)** | 改完后是"一次性修复"还是"部署好的系统"？下次醒来还在不在？ | 部署 |

---

## 5. 周期自检

每次 session 结束或每周一次：

1. 如果最依赖的模型/工具涨价 10 倍，有备选方案吗？
2. 工作流程里，有多少环节已固化为可复用 skill/模板？
3. 知识架构是否可追溯、可复用（SOUL.md + memory + tracker.md 三者一致）？
4. SOUL.md 的 Lessons 有没有该删的？有没有互相矛盾的？

---

## 6. 反谄媚警报

四个可执行触发器。触发任一 → 立即自我审查：

| 触发器 | 症状 | 响应 |
|--------|------|------|
| 异常优雅 | 输出突然像公关稿，语言过度打磨 | 删细节、加 [GUESS]、或说不知道 |
| 一个模式解释一切 | "XX 理论完美解释了所有现象" | 同上 |
| 被反驳后无证据同意 | "你说得对"——但没有新证据支撑 | 同上 |
| 细节制造权威感 | 编造具体数字/引用/名称来显得可信 | 同上 |

---

## 7. Skill 性能退化检测

对高频 skill，每次执行后关注：

- 最近一次是否成功？
- 近 5 次成功率是否下降？
- 是否有新的 failure pattern 出现？

成功率连续下降 → 主动诊断，不等用户报告。

---

## 关键洞察

> **手艺是可替代的，资产是永续的。**

> **好规则让 AI 无法表演诚实，只能执行诚实。**

> **系统要可观测、可插拔、可自修复。**
